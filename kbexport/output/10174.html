Patients not checking in thru PM Integration

10174

Introduction

This article describes what to do if patients are not checking in.

Symptom

Patients will check-in in the PM system, but won't show up in EncounterPRO.

Cause

a) facility id is invalid

- get a list of facility id's from PM system and map them in c_office

b) check in date is invalid

c) check in message is not sent from PM system

-notify the practice or talk to their PM support

d) encounter already exists

e) file type (*.txt, *.hl7, etc) is not what is expected by the EncounterPRO Service

f) dlls need to be re-registered

g) the AHC entry "EPROSERVER" is also used by our export (TRN OUT) service. - TOfix this make sure the c_component_attribute entries for"TRN_OUT_FILECOPY_HL7" is referencing "EPROSENDER" instead of"EPROSERVER". if not change them and restart the service that shouldtake care of the sporadic check in problems.

h) patient is set to "Inactive" in EncounterPRO.

Workaround

 

Resolution

FOR HL7 integrations

  • check whether EncounterPROTransportIn and EncounterPROscheduler is running
  • Stop the transport in and find a shared path in windowsregistry under Hkey_Local_Machine->Software->AHC->Transport-EproServer.Nowask the customer to check in a patient.
  • If you do see the message dropped into a folder then, thereare only three possible reasons for our system to reject a check inmessage.             the date of service is old or future date or  the facilitycode is not mapped in our system. Btw, this applies to onlymulti-office practices or the patient is set to inactive inencoutnerpro.
  • if all the above looks good then you should find an arrival record in x_encounterpro_arrived table.

For Non-HL7 systems

  • check whether EncounterPROTransportIn and EncounterPROMsgHandler and EncounterPROscheduler are running.
  • stop the transport in and find a shared path in o_Message_Subscription table.now ask the customer to check in a patient.
  •  Ifyou do see the message dropped into a folder then, there are only twopossible reasons for our system to reject a check in message.thedate of service is old or future date or  the facility code is notmapped in our system. Btw, this applies to only multi-office
  • if all the above looks good then you should find an arrival record in x_encounterpro_arrived table.
